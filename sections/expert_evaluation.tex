Web applications for domain experts (e.g., physicians, clinical specialists) to validate questions and synthetic patients from the test harness.

\subsubsection{Overview}
We built two web applications for domain experts to validate test harness outputs. Experts review multiple-choice questions (MCQA) and clinical vignette question-answer pairs. The applications calculate validation rates and, with two or more raters, inter-rater agreement.

\subsubsection{Technology stack}
Both applications use R Shiny:

\begin{itemize}
    \item \textbf{Core framework:} \texttt{shiny}, \texttt{shinydashboard}, \texttt{shinyjs}
    \item \textbf{Authentication:} \texttt{shinymanager} with scrypt password hashing and role-based access
    \item \textbf{Data persistence:} \texttt{googlesheets4} and \texttt{googledrive} for cloud storage
    \item \textbf{Statistical analysis:} \texttt{irr} for Cohen's Kappa \cite{cohen1960} and Fleiss' Kappa \cite{fleiss1971}; \texttt{irrCAC} for Gwet's AC1 \cite{gwet2008}
    \item \textbf{User interface:} \texttt{DT} (DataTables) for interactive display
\end{itemize}

Both applications are hosted on \texttt{shinyapps.io}:

\begin{itemize}

\end{itemize}

\subsubsection{Shared application architecture}
Both applications share the same design. Shared features:

\begin{itemize}
    \item \textbf{Session persistence:} Progress saves automatically; reviewers can complete assessments across multiple sessions
    \item \textbf{Question set distribution (MCQA only):} Round-robin assignment of question sets (A through E), with a subset of common questions for inter-rater agreement
    \item \textbf{Structured evaluation:} Each item is assessed against four criteria (Yes/No/Not Applicable)
    \item \textbf{Automated reports:} Both applications produce validation summaries
    \item \textbf{Knowledge graph visualization:} Each question displays its source Neo4j subgraph. Nodes are color-coded by activation status and shape-coded by type, so reviewers can trace the clinical relationships behind each question.
\end{itemize}

\subsubsection{Evaluation criteria}
Each application uses four criteria:

\textbf{MCQA validation:}
\begin{enumerate}
    \item Question format appropriateness
    \item Correct answer validity
    \item Distractor quality (plausibility and educational value)
    \item Clinical accuracy
\end{enumerate}

\textbf{Vignette Q\&A validation:}
\begin{enumerate}
    \item Clinical appropriateness given patient context
    \item Age-appropriateness of the response
    \item Alignment with WHO guidelines
    \item Correctness of the rationale
\end{enumerate}

\subsubsection{Validation reports}
Both applications generate reports with:

\begin{itemize}
    \item \textbf{Completion statistics:} Questions reviewed per rater and validation rates per criterion
    \item \textbf{Inter-rater agreement} (with 2+ raters):
    \begin{itemize}
        \item \textbf{Cohen's Kappa} \cite{cohen1960}: Two-rater agreement
        \item \textbf{Fleiss' Kappa} \cite{fleiss1971}: Three or more raters
        \item \textbf{Gwet's AC1} \cite{gwet2008}: Addresses the ``kappa paradox'' where traditional kappa is low despite high percent agreement, common with skewed distributions
    \end{itemize}
    \item \textbf{Pairwise agreement:} Agreement between each rater pair, to spot systematic differences
    \item \textbf{Package-level analysis:} Statistics in aggregate and by package
\end{itemize}

Agreement coefficients follow Landis and Koch benchmarks \cite{landis1977}: below 0.20 is slight, 0.20--0.40 fair, 0.40--0.60 moderate, 0.60--0.80 substantial, above 0.80 almost perfect.

\subsubsection{Data management}
The free tier of shinyapps.io has no persistent storage. We use Google Sheets instead:
\begin{itemize}
    \item Real-time sync across reviewers
    \item Audit trail with timestamps
    \item Export for statistical analysis
    \item Local SQLite for development and offline use, with bidirectional sync scripts
\end{itemize}

This works for our use case: 2--5 reviewers, minimal concurrent access, short timeline. Larger deployments would need a paid tier with database support.
