\subsubsection{Overview}
We developed two web applications for domain experts to validate the test harness outputs. Experts review multiple-choice questions (MCQA) and clinical vignette question-answer pairs. The applications calculate validation rates (the primary metric) and, when two or more raters are involved, inter-rater agreement to check whether reviewers agree with each other.

\subsubsection{Technology Stack}
Both applications are built with R Shiny:

\begin{itemize}
    \item \textbf{Core Framework:} \texttt{shiny}, \texttt{shinydashboard}, \texttt{shinyjs}
    \item \textbf{Authentication:} \texttt{shinymanager} with scrypt password hashing for user authentication and role-based access control
    \item \textbf{Data Persistence:} \texttt{googlesheets4} and \texttt{googledrive} for cloud storage of responses
    \item \textbf{Statistical Analysis:} \texttt{irr} for Cohen's Kappa \cite{cohen1960} and Fleiss' Kappa \cite{fleiss1971}; \texttt{irrCAC} for Gwet's AC1 \cite{gwet2008}
    \item \textbf{User Interface:} \texttt{DT} (DataTables) for interactive data display
\end{itemize}

\subsubsection{Shared Application Architecture}
Both applications share the same design and interface. Shared features:

\begin{itemize}
    \item \textbf{Session Persistence:} Progress saves automatically, so reviewers can complete assessments across multiple sessions
    \item \textbf{Question Set Distribution (MCQA only):} Round-robin assignment of question sets (A through E), with a subset of common questions across all sets for inter-rater agreement calculations
    \item \textbf{Structured Evaluation Criteria:} Each item is assessed against four criteria using Yes/No/Not Applicable responses
    \item \textbf{Automated Report Generation:} Both applications produce validation summary reports
    \item \textbf{Knowledge Graph Visualization:} Each question displays its source Neo4j subgraph, showing how the question was generated from the knowledge graph. Nodes are color-coded by activation status and shape-coded by type, so reviewers can trace the clinical relationships used to construct each question.
\end{itemize}

\subsubsection{Evaluation Criteria}
Each application uses four criteria:

\textbf{MCQA Validation Criteria:}
\begin{enumerate}
    \item Question format appropriateness
    \item Correct answer validity
    \item Distractor quality (plausibility and educational value)
    \item Clinical accuracy
\end{enumerate}

\textbf{Vignette Q\&A Validation Criteria:}
\begin{enumerate}
    \item Clinical appropriateness given the patient context
    \item Age-appropriateness of the response
    \item Alignment with WHO guidelines
    \item Correctness of the provided rationale
\end{enumerate}

\subsubsection{Validation Reports}
Both applications generate validation reports with:

\begin{itemize}
    \item \textbf{Completion Statistics:} Number of questions reviewed per rater, overall progress, and validation rates (percentage marked valid for each criterion)
    \item \textbf{Inter-Rater Agreement Metrics} (when 2+ raters):
    \begin{itemize}
        \item \textbf{Cohen's Kappa} \cite{cohen1960}: Agreement between two raters
        \item \textbf{Fleiss' Kappa} \cite{fleiss1971}: Agreement among three or more raters
        \item \textbf{Gwet's AC1} \cite{gwet2008}: Addresses the ``kappa paradox'' where traditional kappa yields low values despite high percent agreement, common with skewed distributions
    \end{itemize}
    \item \textbf{Pairwise Agreement Analysis:} Agreement between each pair of raters, to identify systematic differences among reviewers
    \item \textbf{Package-Level Analysis:} Statistics reported in aggregate and by package
\end{itemize}

Agreement coefficients are interpreted using the Landis and Koch benchmarks \cite{landis1977}: values below 0.20 indicate slight agreement, 0.20--0.40 fair agreement, 0.40--0.60 moderate agreement, 0.60--0.80 substantial agreement, and values above 0.80 indicate almost perfect agreement.

\subsubsection{Data Management}
The free tier of shinyapps.io does not support persistent storage or database writes. We use Google Sheets as the primary data store instead:
\begin{itemize}
    \item Real-time synchronization across reviewers
    \item Audit trail with timestamps
    \item Export for statistical analysis
    \item Local SQLite database for development and offline use, with scripts to sync bidirectionally with Google Sheets
\end{itemize}

This approach is acceptable for our use case: only 2--5 expert reviewers, minimal concurrent access, and a short project timeline. For larger-scale deployments, a paid hosting tier with database support would be more appropriate.
